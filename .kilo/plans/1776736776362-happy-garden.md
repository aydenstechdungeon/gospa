## Phase Plan

## Phase 1 - Emergency Security + Correctness Hotfixes
**Goal:** Remove exploitable paths and severe correctness failures first.

### Scope
- SEC-01, SEC-02, REL-01, REL-02.

### Implementation tasks
1. Harden `/islands/*` path handling:
   - Normalize with `filepath.Clean`.
   - Reject traversal or absolute escape attempts.
   - Enforce base-directory containment via absolute-path prefix guard.
   - Return explicit 400/403 statuses.
2. Secure CSRF token injection:
   - Validate token format (strict allowlist, expected length/charset).
   - JSON-encode token before embedding in script context.
   - Ignore/clear invalid cookie values.
3. Split SafeMode validators by language:
   - Keep Go validator for Go snippets.
   - Add TS-aware validator path for `<script lang="ts">`.
   - Ensure TS syntax is not parsed by Go AST parser.
4. Reject trailing JSON payload data in remote parser:
   - Decode first value.
   - Require EOF after decode.

### Verification
- Add focused unit tests for each fix (including negative tests and PoCs).
- Add regression fixtures for TS SafeMode acceptance.
- Run targeted package tests plus repo test pass.

### Exit criteria
- PoCs for traversal/XSS/trailing-garbage fail safely.
- TS SafeMode valid samples compile.
- No behavior regressions in existing security middleware paths.

## Phase 2 - Auth Surface + Path Safety Hardening
**Goal:** Eliminate medium-risk auth and local file-write safety issues.

### Scope
- SEC-03, SEC-04, SEC-05, REL-03.

### Implementation tasks
1. OTP verify hardening:
   - Require authenticated principal.
   - Resolve OTP secret from server-side user record only.
   - Preserve or strengthen per-user+IP rate limiting and lockout semantics.
   - Return non-enumerable error responses.
2. Codegen output confinement:
   - Centralize path sanitizer utility for plugins.
   - Enforce workspace-root containment with absolute checks.
   - Reject traversal/absolute output unless explicit escape flag is configured.
3. SSE client ID integrity:
   - Prefer server-minted IDs.
   - If client IDs remain supported, bind to auth principal and reject collisions.
4. SSE lock scope correction:
   - Snapshot recipients under lock, send outside lock.

### Verification
- Auth integration tests for OTP flow and brute-force controls.
- Path traversal-style tests for codegen output destinations.
- SSE concurrency tests for connect/disconnect/broadcast race scenarios.

### Exit criteria
- OTP endpoint unusable without valid auth.
- No out-of-project writes without explicit opt-in.
- SSE map integrity maintained under collision attempts and high churn.

## Phase 3 - Compiler + Router + Realtime Performance Sprint
**Goal:** Reduce CPU/alloc hotspots and contention.

### Scope
- PERF-01..PERF-08 (grouped by subsystem).

### Implementation tasks
1. Compiler parser efficiency:
   - Precompute newline offsets (or incremental cursor) for offset->position.
   - Hoist regexes to package-level compiled vars.
   - Add SafeMode expression validation caching/prefiltering.
2. Router dynamic matching:
   - Precompile route pattern segments.
   - Reduce per-request splitting/map allocations.
   - Evaluate trie/index path for dynamic routes.
3. WebSocket/SSE throughput hardening:
   - Remove close-under-lock in WSHub reconnect path.
   - Replace unbounded goroutine fallback with bounded backpressure strategy.
   - Maintain topic index (`clientsByTopic`) for SSE fanout.
4. Render alloc reductions:
   - Reuse props maps where safe.
   - Remove duplicate matching/reconstruction in render path.

### Verification
- Add microbenchmarks for compiler/router/realtime critical functions.
- Capture before/after benchmarks (`benchstat` style).
- Load-test reconnect storms and topic fanout paths.

### Exit criteria
- Measurable gains aligned with audit targets.
- No new data races (`-race`) or correctness regressions.

## Cross-Phase Standards

- One PR per phase (or per subsystem in Phase 3 if too large).

### Test strategy

- Concurrency fixes require stress/race tests.
- Perf fixes require benchmark artifacts before merge.

### Release strategy
- After Phase 1: patch release recommended.
- After Phase 2-3: minor release with perf notes.
- After Phase 4-5: docs/security baseline release with CI policy announcement.
