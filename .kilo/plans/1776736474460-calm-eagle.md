# GoSPA Remediation Plan - Multi-Phase Execution

## Goals
- Fix high-risk security and correctness issues first, then stabilize performance and docs.
- Group work into large, executable phases with clear entry/exit criteria.
- Minimize regressions with test and CI hardening embedded in each phase.

## Phase 0 - Triage, Branching, and Safety Net (Preparation)
**Scope**
- Confirm current baseline: failing/passing tests, lint, build, benchmarks.
- Create issue tracker board grouped by Security, Reliability, Performance, Docs, CI.
- Add temporary feature flags or guarded rollouts where behavior changes could break users.

**Deliverables**
- Baseline report: test pass rate, benchmark snapshots, known failing tests.
- Tracking matrix mapping each finding to owner, file, risk, and target phase.

**Exit Criteria**
- Repro steps exist for each High/Medium finding.
- Baseline metrics captured for compiler, routing, SSE/WS, SSR render path.

---

## Phase 1 - Critical Security Patches (Immediate)
**Priority:** Highest (same-day/next-day)

**Scope**
- Path traversal hardening in dynamic `/islands/*` serving (`gospa.go`).
- CSRF token inline script hardening (`fiber/middleware.go`): strict token validation + JSON encoding before JS injection.
- OTP verify endpoint tightening (`plugin/auth/auth.go`): require authenticated principal and server-side secret binding.
- SSE client identity hardening (`fiber/sse.go`): prevent caller-chosen ID collisions/overwrite.

**Implementation Notes**
- Use absolute-path containment checks (`filepath.Abs` + base-prefix guard).
- Reject malformed/unsafe token values, do not silently reflect raw cookie values into script context.
- Keep rate limit at per-user+IP once auth enforcement is added.

**Validation**
- Security regression tests for traversal, injected cookie payloads, unauth OTP calls.
- Manual safe PoC verification shows blocked behavior.

**Exit Criteria**
- All High security findings are patched and covered by tests.
- No route can escape `generated/` path.
- No inline JS concatenates untrusted raw token values.

---

## Phase 2 - Compiler Correctness and Reliability
**Priority:** High

**Scope**
- Split SafeMode validators by language (`Go` vs `TS`) in `compiler/compiler.go`.
- Ensure `.gospa` `<script lang="ts">` works with SafeMode enabled.
- Fix `remote_input.go` multi-root/trailing-data JSON acceptance.
- Add fuzz targets and seed corpus for SFC template parser and JSON decode paths.

**Implementation Notes**
- Introduce dedicated TS-safe validation path that does not use Go parser.
- Enforce strict JSON EOF semantics after first decode.

**Validation**
- New unit tests for SafeMode TS/Go positive and negative cases.
- Fuzz smoke runs in CI (time-bounded) with reproducible corpus.

**Exit Criteria**
- SafeMode no longer rejects valid TS due to Go parser mismatch.
- Remote JSON parser rejects trailing garbage deterministically.
- Fuzz targets run and produce no immediate crashes/panics.

---

## Phase 3 - Dependency and Supply-Chain Remediation
**Priority:** High

**Scope**
- Resolve JS advisories across `client/`, `website/`, `vscode-extension/`, prioritizing critical `minimist` chain.
- Update Go module for `golang.org/x/image/tiff` advisory path where applicable.
- Remove/replace vulnerable transitive deps where updates are unavailable.

**Implementation Notes**
- Prefer lockfile refresh plus direct version pinning for stubborn transitive chains.
- Validate extension packaging/runtime compatibility after updates.

**Validation**
- `bun audit` clean or documented accepted residual risk with expiration.
- `govulncheck` clean for reachable paths.

**Exit Criteria**
- No critical/high dependency advisories remain unaddressed.
- Build/test succeeds in all affected JS packages and Go modules.

---

## Phase 4 - Performance and Concurrency Sprint
**Priority:** Medium-High

**Scope**
- Compiler perf: newline index for `OffsetToPosition`, regex hoisting, SafeMode validation caching.
- Router perf: precompiled dynamic route segments/trie-like matcher to reduce per-request allocations.
- WebSocket/SSE contention: avoid closing sockets while lock held; snapshot fanout recipients and send outside locks.
- Render path alloc reduction: avoid repeated map rebuilds in SSR.

**Implementation Notes**
- Implement in isolated PR slices to reduce blast radius:
  1) compiler hot path,
  2) router matcher,
  3) WS/SSE locking,
  4) SSR alloc cleanup.

**Validation**
- Benchmarks before/after for compile throughput, route QPS, WS reconnect p99, SSE fanout latency.
- Race detector runs for WS/SSE changes.

**Exit Criteria**
- Demonstrated measurable wins against baseline (target ranges from audit).
- No regressions in routing semantics or event delivery.

---

## Phase 5 - Docs and Information Architecture Repair
**Priority:** Medium

**Scope**
- Fix broken links and route mismatches across `README.md`, `docs/**`, `website/routes/docs/**`.
- Unify Go version requirements and establish docs source-of-truth policy.
- Regenerate/fix `website/static/docs_search_index.json` dead URLs.

**Validation**
- Automated link check over repo docs and website routes.
- Spot-check critical onboarding pages and nav flows.

**Exit Criteria**
- No broken internal docs links in CI.
- README and docs version requirements are consistent.

---

## Phase 6 - CI Security/Quality Gates and Release Hardening
**Priority:** Medium

**Scope**
- Add required CI gates: `govulncheck`, `bun audit` (all JS workspaces), link checker, targeted benchmarks/smoke perf checks.
- Add security regression test suite (path traversal, inline token injection, auth guard enforcement).
- Release checklist and rollback plan for security/perf patches.

**Validation**
- CI fails on new critical findings or broken docs links.
- Nightly scheduled scans produce artifacts and triage tickets automatically.

**Exit Criteria**
- Security and documentation drift is continuously enforced.
- Release pipeline has explicit guardrails for regressions.

---

## Cross-Phase Governance
- **Change Management:** one phase per PR series, small mergeable units, explicit migration notes for behavioral changes.
- **Testing Policy:** every vulnerability fix must include a regression test.
- **Performance Policy:** every optimization change must ship with benchmark evidence.
- **Documentation Policy:** every API/behavioral change updates docs in the same PR.

## Suggested Milestone Cadence
- **Milestone A (Week 1):** Phases 0-1 complete.
- **Milestone B (Week 2):** Phase 2 + high-priority Phase 3 updates.
- **Milestone C (Week 3):** Phase 4 in slices.
- **Milestone D (Week 4):** Phases 5-6 and release hardening.

## Definition of Done (Program-Level)
- All High findings closed and verified.
- Medium findings either closed or risk-accepted with documented expiry.
- CI gates enforce security, dependency, and docs quality going forward.
- Benchmarks show net positive performance outcomes without correctness regressions.
