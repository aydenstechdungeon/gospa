# Plan: Usage-Aware Critical CSS Splitting

## Problem

The `splitCSS` function in `plugin/postcss/postcss.go` splits compiled CSS at a naive byte-size boundary. Tailwind v4 outputs utility classes alphabetically within `@layer utilities`, so classes like `bg-gradient-to-br`, `bg-clip-text`, and `text-transparent` fall past the 40KB cutoff and end up in non-critical CSS. This causes LCP elements that depend on these classes to be invisible until the async stylesheet loads, producing inconsistent LCP times (0.15s cached vs 0.5s cold).

## Goal

Make `splitCSS` aware of which CSS rules are used by above-the-fold templates, so those rules are always included in the critical CSS output — fixing LCP for every GoSPA project using the critical CSS feature.

### Implementation Steps

#### Step 1: Class Extraction from Templates (`extractCriticalClasses`)

New function: `extractCriticalClasses(projectDir string, contentPaths []string) map[string]bool`

**Logic:**
1. Resolve glob patterns to file paths
2. For each `.templ` file, read content and extract `class="..."` attribute values
3. Split each class string by whitespace
4. Return deduplicated set of class names

**Regex for class extraction:**
- Match `class="..."` and `class={...}` (templ syntax)
- For string literals: split by whitespace, clean up template expressions
- Return raw class names as they appear in templates

**Example input → output:**
```html
class="bg-gradient-to-br from-[var(--accent-primary)] bg-clip-text text-transparent"
→ {"bg-gradient-to-br", "from-[var(--accent-primary)]", "bg-clip-text", "text-transparent"}
```

#### Step 2: CSS Selector Matching (`selectorMatchesClass`)

Convert a Tailwind class name to its CSS selector form, then check if a CSS rule's selector matches.

**Tailwind v4 class → selector mapping:**
- Simple: `bg-clip-text` → `.bg-clip-text`
- Arbitrary values: `from-[var(--accent-primary)]` → `.from-\[var\(--accent-primary\)\]`
- Responsive: `md:text-8xl` → wrapped in `@media (width >= 48rem)` with selector `.md\:text-8xl`
- Opacity: `bg-white/5` → `.bg-white\/5`
- Pseudo: `hover:bg-red-500` → `.hover\:bg-red-500:hover`

**Approach:** Rather than regex-matching selectors against class names, convert each critical class name to its escaped CSS selector form, then check if that selector appears in the CSS rule block. This is simpler and more reliable.

**Escaping function:** Escape `[`, `]`, `(`, `)`, `:`, `/`, `%` with `\` for CSS selector matching. Tailwind's escaping is consistent and predictable.

#### Step 3: Enhanced `splitCSS` Function

Replace the current byte-boundary-only split with a usage-aware split:

```
func splitCSS(fullCSS []byte, maxSize int, criticalClasses map[string]bool) ([]byte, []byte)
```

**Algorithm:**

1. Parse CSS into top-level blocks, tracking:
   - Block type: `@layer`, `@media`, `@supports`, or flat rule
   - Content within each block
   - Byte positions

2. Classify each block:
   - **Always-critical**: `@layer theme`, `@layer base`, `@layer components`, `@font-face`, `@layer properties`
   - **Critical utility**: An `@layer utilities` block (or sub-block within it) that contains a selector matching a critical class
   - **Non-critical utility**: Everything else in utilities

3. Reconstruct CSS in order:
   - Always-critical blocks first (preserving original order)
   - Critical utilities second
   - Non-critical utilities last

4. Split at byte boundary on the reconstructed CSS, ensuring we don't split mid-rule.

**Detailed parsing approach:**

The compiled CSS is **minified** (all on one line). The minified structure is:
```
@layer properties;@layer theme, base, components, utilities;@layer theme{:root{...}}@layer base{body{...}...}@layer utilities{.class-a{props}@supports(...){props}.class-b{props}@media(...){&:hover{props}}.class-c{props}...}
```

Key insight: `@supports` and `@media` blocks are **nested inside** top-level utility rules, not at the same level. Each top-level rule ends when braceDepth returns to 0. This makes parsing straightforward.

Example minified rule with nested @supports:
```
.bg-white\/5{background-color:color-mix(in srgb,#fff 5%,transparent);@supports (color:color-mix(in lab,red,red)){background-color:color-mix(in oklab,var(--color-white) 5%,transparent)}}
```

**Parsing approach:**
1. Find `@layer utilities{` in the minified CSS
2. Extract the content between the opening `{` and its matching `}` (braceDepth=0)
3. Split the content into individual rules by tracking brace depth:
   - Start a new rule when we see a `.` at braceDepth=0 (after previous rule ended)
   - Rule ends when braceDepth returns to 0
4. For each rule, extract the selector (everything before the first `{`)
5. Check if selector matches a critical class (using `classToSelector` conversion)

**Reconstruction strategy:**
1. Split the full CSS into: prefix (theme + base + components layers) and utilities layer
2. Parse the utilities layer into individual rules
3. Tag each rule as critical (selector matches) or non-critical
4. Reconstruct: prefix + `@layer utilities{` + critical rules + non-critical rules + `}`
5. Apply byte-boundary split on the reconstructed CSS

#### Step 4: Wire Up Configuration

In `criticalCommand` and `extractCriticalForBundle`:
1. Read `CriticalContent` from config
2. Call `extractCriticalClasses` to get the class set
3. Pass the class set to `splitCSS`

If `CriticalContent` is empty, fall back to the current behavior (byte-size-only split). This ensures backward compatibility.

#### Step 5: Tests

Create `plugin/postcss/postcss_test.go` with:

1. **`TestExtractCriticalClasses`**: Verify class extraction from templ files with various syntaxes
2. **`TestClassToSelector`**: Verify CSS selector escaping
3. **`TestSplitCSS_ClassAware`**: Verify that critical classes are prioritized in the split
4. **`TestSplitCSS_BackwardCompat`**: Verify existing behavior when no critical classes are provided
5. **`TestSplitCSS_EdgeCases`**: Empty CSS, CSS smaller than maxSize, no utilities layer, etc.

## Backward Compatibility

- `CriticalContent` is optional. If empty/nil, `splitCSS` behaves exactly as before (byte-size split).
- Existing `CriticalCSSConfig` YAML configs continue to work unchanged.
- No API changes to `CriticalCSS()`, `AsyncCSS()`, `CriticalCSSWithFallback()`.

## CSS Format Note

The compiled CSS output (`website.css`) is **minified** by cssnano — all on one line, no whitespace. The `splitCSS` function splits this minified byte stream directly. The critical/non-critical output files are also minified (the existing pretty-printed files in the repo were generated from a non-minified source before cssnano was enabled).

In the minified format, `@supports` and `@media` blocks are **nested inside** their parent rules (not at the same brace depth). Example: `.bg-white\/5{background-color:color-mix(...);@supports (color:color-mix(in lab,red,red)){background-color:color-mix(...)}}` — the `@supports` is inside the rule's braces, making the entire rule self-contained.

## Edge Cases

1. **No utilities layer**: If the CSS has no `@layer utilities`, split at byte boundary as before.
2. **CSS smaller than maxSize**: No split needed, return as-is.
3. **Critical classes not found**: Classes used in templates but not in compiled CSS (e.g., dynamic classes) — safely ignored.
4. **`@supports` fallback blocks**: These are paired with their parent rule and moved together.
5. **Custom CSS variables in classes**: `from-[var(--accent-primary)]` → selector `.from-\[var\(--accent-primary\)\]` — handled by escaping function.
6. **Responsive variants**: `md:text-8xl` generates `@media (...) { .md\:text-8xl { ... } }` — selector matching works because the escaped class name appears in the selector.
