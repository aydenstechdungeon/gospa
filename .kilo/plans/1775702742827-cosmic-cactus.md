# Plan: Optimize SPA Navigation Performance for Code-Heavy Doc Pages

## Problem

When navigating to code-heavy doc pages (like the runes/reactivity pages) via SPA navigation, the page feels slower than a full-page refresh. On refresh the page renders almost instantly, but on navigation there's a noticeable delay.

## Root Causes

1. **Idiomorph DOM morphing overhead** â€” Code blocks produce many DOM nodes (`<pre><code>` with multi-line text). Idiomorph must diff every node against the old content. On refresh, the browser just paints â€” no diff needed.

2. **`initNewContent` re-initialization** â€” After morphing, the runtime scans the entire new container for islands, bindings, and event handlers. This runs on every navigation but not on refresh.

3. **`updateActiveLinks()` scans all anchors** â€” Walks every `<a[href]>` in the document on every navigation to update active states.

4. **Synchronous `updateToC()` on `gospa:navigated`** â€” Blocks paint while scanning headings and building the ToC.

5. **Async CSS activation delay** â€” Non-critical CSS (including code block scrollbar styles) activates after navigation, potentially causing a brief unstyled flash.

6. **No server-side syntax highlighting** â€” Code blocks are raw text; any client-side highlighter would add processing time.

## Plan

### Phase 1: Defer Non-Critical Post-Navigation Work (Quick Win)

**File:** `website/static/docs.js`

Wrap `updateToC()` in `requestIdleCallback` inside the `gospa:navigated` handler so it doesn't block the paint of the main content. The ToC can populate a few frames later without the user noticing.

```js
document.addEventListener('gospa:navigated', () => {
    saveSidebarScroll();
    // Defer ToC generation so content paints first
    if ('requestIdleCallback' in window) {
        requestIdleCallback(() => updateToC());
    } else {
        setTimeout(updateToC, 0);
    }
    initAsyncCSS();
    setTimeout(() => { restoreSidebarScroll(); }, 0);
});
```

**Impact:** Content becomes visible faster; ToC fills in shortly after.

### Phase 2: Scope `updateActiveLinks()` to Sidebar Only

**File:** `client/src/navigation.ts`

Currently `updateActiveLinks()` queries `document.querySelectorAll("a[href]")` which walks every link on the page. The sidebar links are the only ones with `data-gospa-active`/`data-gospa-inactive` attributes. Scope the query to the sidebar container.

Change the selector from `a[href]` to `#docs-sidebar a[href]` when inside docs pages, falling back to the current behavior for non-docs pages.

**Impact:** Reduces DOM traversal from O(all links) to O(sidebar links) on docs navigation.

### Phase 3: Optimize `initNewContent` Scoping

**File:** `client/src/navigation.ts`

The `initNewContent(container)` function scans the entire morphed container. After a docs-to-docs navigation, only the content area changed (sidebar is `data-gospa-permanent`). Pass the actual changed element rather than the full container.

In `reconcileDOM`, track which nodes were actually morphed (Idiomorph's `afterNodeMorphed` callback). Then scope `initCriticalContent` and `initDeferredBindings` to only the changed subtree.

Alternatively, since the sidebar has `data-gospa-permanent`, add logic in `initNewContent` to skip subtrees with that attribute.

**Impact:** Avoids re-scanning permanent elements on every navigation.

### Phase 4: Add `loading="lazy"` to Code Block DOM Reduction

**File:** `website/components/code_block.templ`

Add a `data-gospa-morph="inner"` attribute to the `<pre>` element inside code blocks. This tells Idiomorph to replace innerHTML rather than diffing every text node child. Multi-line code blocks become a single text node comparison instead of hundreds.

```html
<pre class="..." data-gospa-morph="inner">
    <code>{ code }</code>
</pre>
```

This requires adding support for `data-gospa-morph` in the Idiomorph callbacks in `navigation.ts` â€” check for this attribute in `beforeNodeMorphed` and return `false` to skip child diffing, then manually replace innerHTML.

**Impact:** Dramatically reduces morphing cost for pages with many/large code blocks.

### Phase 5: Server-Side Syntax Highlighting (Optional, Higher Effort)

**File:** `website/components/code_block.templ`, `go.mod`

Add `alecthomas/chroma` as a dependency. Create a `Highlight(code, language string) string` function that returns syntax-highlighted HTML. Update `CodeBlock` to use it.

This eliminates any client-side processing and means the HTML arrives ready to paint. The tradeoff is slightly larger HTML payloads (but already compressed by gzip/brotli).

**Impact:** Zero client-side highlight processing; better perceived performance.

## Implementation Order

1. **Phase 1** â€” `docs.js` change, ~5 lines. Lowest risk, immediate visible improvement.
2. **Phase 2** â€” `navigation.ts` selector change, ~10 lines. Low risk.
3. **Phase 3** â€” `navigation.ts` scoping logic, ~20 lines. Moderate risk, needs testing.
4. **Phase 4** â€” `code_block.templ` + `navigation.ts` callback, ~15 lines. Moderate risk.
5. **Phase 5** â€” New dependency + component change. Higher effort, defer if earlier phases are sufficient.

## Testing

- Navigate between doc pages with many code blocks (e.g., `/docs/gospasfc/reactivity`) and measure time-to-content via browser DevTools Performance tab.
- Verify sidebar active state updates correctly after Phase 2.
- Verify islands and bindings still initialize correctly after Phase 3.
- Verify code blocks render correctly and copy-to-clipboard still works after Phase 4.
- Test on both initial navigation and subsequent navigations (cache behavior).
