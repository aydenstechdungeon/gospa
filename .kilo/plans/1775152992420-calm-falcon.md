# GoSPA Navigation Optimization Plan

## Overview
Optimize GoSPA's client-side navigation to reduce INP from 160ms to under 100ms and make navigation feel noticeably snappier. The optimizations focus on prefetching aggressiveness, request deduplication, service worker caching, connection awareness, and event handler optimization.

---

## Phase 1: Configuration Tuning (Quick Wins)

### 1.1 Update Navigation Configuration
**File:** `website/static/docs.js` (lines 243-256)
- Increase `ttl` from 45000ms to 60000ms
- Reduce `hoverDelay` from 80ms to 50ms
- Increase `viewportMargin` from 220px to 400px

**Rationale:** These are immediate improvements that require no code changes, just configuration tuning. Longer TTL means prefetched pages stay cached longer, shorter hover delay means faster response to user intent, and larger viewport margin means more links get prefetched proactively.

---

## Phase 2: Request Deduplication

### 2.1 Add Pending Request Cache
**File:** `client/src/navigation.ts`
- Add `const pendingRequests = new Map<string, Promise<PageData | null>>();` near line 156
- Modify `fetchPageFromServer()` function (lines 388-452) to:
  - Check if a request for the same path is already in flight
  - If yes, return the existing promise instead of creating a new fetch
  - Clean up the map entry when the request completes or fails

**Rationale:** Prevents duplicate network requests when multiple prefetch triggers fire for the same page (e.g., IntersectionObserver + hover on the same link).

---

## Phase 3: Connection-Aware Prefetching

### 3.1 Add Network Intelligence
**File:** `client/src/navigation.ts`
- Add `shouldPrefetch()` function that checks `navigator.connection` API
- Returns `false` for slow-2g, 2g, or when `saveData` is enabled
- Update `setupSpeculativePrefetching()` to check `shouldPrefetch()` before setting up observers

**Rationale:** Prefetching on slow connections or metered data hurts performance and user experience. This ensures prefetching only happens when it's beneficial.

---

## Phase 4: Service Worker Cache Strategy

### 4.1 Update Service Worker to Stale-While-Revalidate
**File:** `website/static/gospa-navigation-sw.js`
- Change from network-first to stale-while-revalidate strategy:
  - Return cached response immediately if available
  - Fetch from network in background to update cache
  - Fall back to network only if no cache exists

**Rationale:** Stale-while-revalidate provides instant navigation from cache while keeping content fresh in the background. This is ideal for documentation sites where content doesn't change frequently.

---

## Phase 5: DOM Update Optimization

### 5.1 Defer Non-Critical Post-Navigation Work
**File:** `client/src/navigation.ts`
- In `performDOMUpdateWithTransitions()` (lines 1014-1047):
  - Move `updateActiveLinks()` and other non-critical work into `requestIdleCallback()`
  - Keep critical DOM updates synchronous for visual correctness
  - Set timeout of 2000ms for idle callback to ensure it runs

**Rationale:** Reduces INP by ensuring the main thread isn't blocked by non-critical work immediately after navigation. The user sees the page update instantly, and link highlighting happens during idle time.

---

## Phase 6: Priority-Based Prefetching (Optional Enhancement)

### 6.1 Add Priority Queue System
**File:** `client/src/navigation.ts`
- Add `PrefetchTask` interface with path, priority, and timestamp
- Create `prefetchQueue` Set and processing logic
- Modify `setupSpeculativePrefetching()` to enqueue with priorities:
  - Priority 3: Links in sidebar navigation
  - Priority 2: Links in main content area
  - Priority 1: All other internal links
- Process queue with small delays (50ms) between requests to avoid network congestion

**Rationale:** Ensures the most likely navigation targets are prefetched first, improving perceived performance for the most common user paths.

## Implementation Order

1. **Phase 1** - Configuration tuning (5 min, zero risk)
2. **Phase 2** - Request deduplication (10 min, low risk)
3. **Phase 3** - Connection awareness (10 min, low risk)
4. **Phase 4** - Service worker update (10 min, medium risk - test caching behavior)
5. **Phase 5** - DOM update optimization (10 min, low risk)
6. **Phase 6** - Priority queue (15 min, optional enhancement)

**Total estimated time:** 50-60 minutes

---

## Testing Strategy

1. **Before/After Metrics:**
   - Use Chrome DevTools Performance panel to measure INP
   - Use Lighthouse to measure overall performance scores
   - Test on both fast (desktop WiFi) and slow (throttled 3G) connections

2. **Manual Testing:**
   - Rapid navigation between pages - verify no duplicate requests
   - Hover over links - verify prefetch timing feels responsive
   - Test with service worker enabled/disabled
   - Test on mobile with slow connection - verify prefetching is disabled

3. **Regression Testing:**
   - Verify all existing navigation still works
   - Verify sidebar scroll position preservation
   - Verify ToC updates correctly after navigation
   - Verify language switching still works
