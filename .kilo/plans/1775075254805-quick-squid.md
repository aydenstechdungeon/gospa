# GoSPA DX/UX Improvements Plan

## Overview
This plan implements comprehensive DX (Developer Experience) and UX (User Experience) improvements across GoSPA, with emphasis on the CLI commands (`create`, `dev`, `build`, `doctor`, `generate`) and the remote action system.

---

## Phase 1: CLI Improvements

### 1.1 `gospa doctor` Enhancements

**Current State:**
- Basic binary/path checks (go, bun, go.mod, main.go, etc.)
- No version compatibility checking

**Improvements:**
1. Check `templ` binary version compatibility:
   - `doctor.go`: Add check for `templ` binary via `go install` list
   - Compare against version expected by framework (currently `v0.3.1001`)
   
2. Check `go.mod` replace directives:
   - Detect if user has replace directives that might conflict
   - Warn if gospa is replaced with local path (could cause build issues)

3. Check Bun version:
   - Verify Bun version >= 1.0 (minimum for GoSPA client build)
   - Warn if using Bun via global install vs project-local

4. Check islands.js exists:
   - Verify `/static/js/islands.js` exists or `/generated/islands.ts` was built
   - Currently `render.go:296` hardcodes this path with no warning if missing

5. Check generated runtime matches framework version:
   - Compare `client/dist/runtime.js` version or hash against embedded runtime

**Files to Modify:**
- `cli/doctor.go`

---

### 1.2 `gospa create` Enhancements

**Current State:**
- Creates basic project structure
- No template selection
- Hardcoded templ version in generated files

**Improvements:**
1. Interactive template selection:
   - Add `--template` flag with options: `default`, `minimal`, `api`, `realtime`
   - Templates determine which files are generated (e.g., minimal skips CSS)

2. Validate module path format:
   - The current `getGitUsername()` fallback to "yourusername" can generate invalid module paths
   - Add explicit `--module` flag or prompt for GitHub org/user

3. Create starter Tailwind CSS configuration:
   - Currently `create.go:354` creates raw CSS
   - If `--template` includes Tailwind, generate `postcss.config.js` and `@theme` directives

4. Add starter `tailwind.config.js` support:
   - Follow Tailwind v4 conventions with `@config` in CSS

**Files to Modify:**
- `cli/create.go`
- `cmd/gospa/main.go` (add template flag)

---

### 1.3 `gospa dev` Enhancements

**Current State:**
- Polling-based file watcher (500ms interval in `dev.go:222`)
- Server restart on any `.go` or `.templ` change
- No dev-mode specific optimizations

**Improvements:**
1. Improve file watcher:
   - Use `github.com/fsnotify/fsnotify` instead of manual polling
   - Add debouncing for rapid file changes (100ms)

2. Add dev server optimizations:
   - Enable Fiber's verbose logging in dev
   - Add `--verbose` flag for detailed request logging
   - Add `--no-restart` flag for templates/static-only changes

3. Browser auto-refresh on static file changes:
   - Currently `dev.go:393` just prints "browser will reload"
   - Implement actually triggering reload via WebSocket or SSE

4. Add error overlay integration:
   - Show compile/runtime errors in browser
   - Leverage existing `client/src/error_overlay.ts`

5. Add `--open` flag to automatically open browser

**Files to Modify:**
- `cli/dev.go`
- `cmd/gospa/main.go` (add new flags)

---

### 1.4 `gospa build` Enhancements

**Current State:**
- Builds Go binary, copies static, compresses
- Islands bundle built to `static/js/islands.js`
- No error handling for missing islands.js reference

**Improvements:**
1. Validate islands.js is built before referencing:
   - In `build.go:105-109`, islands build failure is "non-fatal"
   - Add warning if islands.ts exists but build failed
   - Add config option to fail build on islands error

2. Add progress output for large builds:
   - Currently build is silent during compression phase
   - Show spinner/progress for compression step

3. Add build artifacts manifest:
   - Generate `dist/manifest.json` with file hashes
   - Enable cache-busting in production

4. Add source map generation:
   - Pass `--map` flag to Bun build for debugging
   - Add `--no-source-map` to disable

5. Validate output directory is clean:
   - Warn if output dir already exists and has old build

**Files to Modify:**
- `cli/build.go`
- `render.go` (remove hardcoded islands.js path)

## Phase 2: Type-Safe Remote Actions

### 2.1 Codegen for Remote Actions

**Current State:**
- Remote actions registered manually in `routing/remote.go`
- No type safety between Go handler and TypeScript caller

**Improvements:**
1. Generate TypeScript from Go remote action signatures:
   - Parse `routing.RegisterRemoteAction()` calls during generate
   - Extract input/output struct types
   - Generate `generated/remote-actions.ts` with typed wrappers

2. Generate Go typed wrappers:
   - Generate adapter functions that deserialize into typed structs
   - Avoid manual `input.(map[string]any)` assertions

3. Example output:
   ```typescript
   // Generated from Go:
   // routing.RegisterRemoteAction("submitMessage", func(...) (Message, error))
   export const submitMessage = remoteAction<MessageInput, Message>('submitMessage');
   ```

**Files to Modify:**
- `cli/generate.go`
- New file: `routing/generator/remote.go`

---

### 2.2 Remote Action Validation

**Improvements:**
1. Validate action names match in Go and TS:
   - Add `gospa doctor` check for orphaned TS calls
   - Warn if TS calls action that doesn't exist in Go registry

2. Validate input/output types match:
   - Compare Go struct fields to TS interface
   - Warn on mismatches (e.g., Go int vs TS string)

## Phase 3: Config Builder Pattern

### 3.1 Functional Options for Config

**Current State:**
- `Config` struct has 50+ fields
- Users set fields directly on struct
- No discoverability of available options

**Improvements:**
1. Add functional options:
   ```go
   app := gospa.New(
       gospa.DefaultConfig(),
       gospa.WithAppName("myapp"),
       gospa.WithDevMode(true),
       gospa.WithWebSocket(true),
       gospa.WithPort(8080),
   )
   ```

2. Define option functions in `config.go`:
   - `WithAppName(name string) ConfigOption`
   - `WithDevMode(b bool) ConfigOption`
   - `WithPort(port string) ConfigOption`
   - `WithWebSocketPath(path string) ConfigOption`
   - etc.

3. Update `DefaultConfig()` to return functional config
   - Keep backward compatibility by making struct field writes work after options

**Files to Modify:**
- `config.go`

## Phase 4: Streaming UX Improvements

### 4.1 Streaming Error Handling

**Current State:**
- `Suspense` component in `streaming.go:433` streams errors to console only
- Users see no visual feedback on failure

**Improvements:**
1. Add configurable error fallback:
   ```go
   templ.Suspense(loader, fallback, SuspenseOptions{
       ErrorFallback: func(err error) templ.Component {
           return ErrorComponent(err.Error())
       },
       Retry: true,
   })
   ```

2. Add visible error state in streaming handler:
   - Show error in place of loading content
   - Add "Retry" button

**Files to Modify:**
- `templ/streaming.go`

## Phase 5: Hardcoded Path Fixes

### 5.1 Remove Hardcoded islands.js Path

**Current Issue:**
- `render.go:296` hardcodes `<script src="/static/js/islands.js"></script>`
- No check if file exists

**Fix:**
1. Check for islands bundle existence:
   - If `static/js/islands.js` or `generated/islands.ts` exists, include script
   - Otherwise skip (islands are optional)

2. Make path configurable:
   - Add `Config.IslandsBundlePath` field
   - Default to `/static/js/islands.js`

**Files to Modify:**
- `render.go`
- `config.go`

## Phase 6: Error Message Improvements

### 6.1 DevMode Error Exposure

**Current State:**
- `gospa.go:406` returns "Internal server error" to client even in DevMode
- Developers must check server logs

**Improvements:**
1. In DevMode, include error in response:
   ```json
   {
     "error": "invalid input",
     "code": "ACTION_FAILED",
     "debug": "Error message here" // Only in DevMode
   }
   ```

2. Add structured error codes:
   - Map Go errors to client-friendly codes
   - Document codes in error messages

**Files to Modify:**
- `gospa.go`
- `client/src/remote.ts`
